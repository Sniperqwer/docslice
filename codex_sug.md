# Codex 建议（第一次：针对 `tech_spec.md` 的完整改进意见）

我这次不是只挑几个大方向，而是按 PRD 的产品目标把 `tech_spec.md` 整体过了一遍，尽量把所有我认为有改进空间、且会影响 V1 路径质量的点都列出来。总判断先放前面：

**当前 `tech_spec.md` 的问题，不是“方向错了”，而是更偏工程实现草案，还没有完全围绕 PRD 的核心价值收敛。**  
PRD 的核心不是“做一个网页转 Markdown 工具”，而是：

- 生成可裁剪的蓝图
- 保住目录层级
- 输出干净、稳定、适合喂给 LLM 的切片
- 为后续增量同步留空间

所以判断一条技术路线是否合适，标准不该是“能不能跑通”，而该是“是否稳定、可复现、可调试，并且不背离 PRD 的设计原则”。

---

## 1. 顶层技术决策写得太快，缺少和 PRD 的对应关系

开头写了：

- `typer` CLI
- V1 仅支持静态 HTML
- 目标发布方式 `brew install`

这里最大的问题不是技术选型本身，而是这些决策没有按 PRD 优先级排序。

### 我建议的改法

- `typer` 可以保留，没问题。
- “V1 仅支持静态 HTML”要写成**范围约束**，而不是像产品能力声明。
- “目标发布方式 `brew install`”不应该放在开头作为同等重要的技术决策。

### 原因

PRD 的第一优先级是 `gen -> 手动裁剪 -> fetch` 这条链路稳定，不是安装分发方式。把 `brew install` 放得太靠前，会误导实现节奏，容易把注意力过早分散到打包和分发，而不是抓取质量和蓝图体验。

---

## 2. V1 Scope 的定义还不够“产品化”

现在 V1 Scope 只写了“做 `gen` + `fetch`，其他延后”。这还不够，因为 PRD 对 V1 的要求其实不是“两个命令存在”，而是“这两个命令的行为必须符合设计理念”。

### 建议补充 V1 的显式目标

V1 的 scope 应补成下面这种表达：

- `gen` 必须产出**人类可编辑、可复用、可复现**的蓝图
- `fetch` 必须产出**干净、编号稳定、层级语义清晰**的 Markdown 集合
- V1 默认只支持**静态 HTML 文档站**
- V1 不追求全站自动识别覆盖率，优先保证少量常见站点的质量

### 原因

PRD 的设计原则是“宁可少自动，也不要错误自动”“干净优先”“渐进增强”。现在 spec 的 scope 还是偏功能项罗列，没有把这些原则翻译成技术约束。

---

## 3. 项目结构拆分还不够贴合后续演进，尤其缺少清洗层和适配层

当前结构：

- `generator.py`
- `fetcher.py`
- `parser.py`
- `converter.py`
- `presets.py`

这对 demo 足够，但对 PRD 的中期路线不够稳。

### 主要问题

- `parser.py` 同时承担 TOC 提取和正文提取，职责偏大。
- `converter.py` 只写“封装 markdownify”，但 PRD 的重点其实是“干净优先”，不是“转换器调用成功”。
- `presets.py` 更像 selector 配置表，不像站点适配层。
- 没有 manifest / metadata 层，导致后续 sync 会比较难接。

### 我建议的目录方向

至少可以演进成：

- `adapters.py` 或 `adapters/`
- `generator.py`
- `fetcher.py`
- `content_extractor.py`
- `cleaner.py`
- `renderer.py`
- `manifest.py`
- `models.py`
- `http.py` 或 `client.py`
- `naming.py`

### 原因

PRD 后面明确要做 sync、图片本地化、metadata/frontmatter。当前结构如果不提前把“抓取”“提取”“清洗”“渲染”“状态记录”拆开，V1.5 基本一定会回头拆主干。

---

## 4. `cli.py` 的命令设计还不够完整，尤其是参数职责不清

当前命令：

```bash
docslice gen <url> [--toc-selector "nav ul"] [--preset docusaurus]
docslice fetch [--output ./output] [--force] [--delay 1.5]
```

### 我认为有改进空间的点

- `fetch --delay` 和蓝图里的 `config.delay` 谁优先，没有定义。
- `fetch --force` 在 V1 没有 sync 的情况下，语义其实不清。
- `gen` 只有 `--toc-selector`，但正文 selector 的来源没有对用户暴露。
- 缺少 dry-run / validate 类能力，用户编辑完 YAML 后没有检查手段。

### 建议

- 明确 CLI 优先级：命令行参数覆盖蓝图配置。
- `fetch --force` 如果 V1 没有缓存或输出跳过逻辑，这个参数没有必要；要么删掉，要么明确语义为“覆盖已有输出目录中的同名文件”。
- `gen` 建议允许 `--content-selector`，否则用户只能依赖 preset 猜中正文。
- 增加 `docslice validate`，至少做：
  - YAML 结构校验
  - URL 规范化检查
  - 重复 URL 检查
  - 缺失标题 / 缺失 URL 的节点提示

### 原因

PRD 的核心工作流里最重要的一步是“手动裁剪”。如果用户编辑完蓝图后没有验证手段，这条工作流会变得很脆，和 PRD 想强调的“可控性”不完全一致。

---

## 5. 蓝图设计还不够支撑“可复现执行”

当前蓝图只存：

- `project_name`
- `base_url`
- `config.selector`
- `delay`
- `toc`

### 主要问题

- `selector` 这个字段命名不清，到底是 TOC selector 还是 content selector？
- `gen` 时做出的关键判断没有落盘，导致 `fetch` 可复现性不足。
- 没有版本字段，后面蓝图结构演进会难做兼容。

### 建议至少补这些字段

```yaml
version: 1
project_name: "claude_code_docs"
base_url: "https://docs.anthropic.com"
config:
  adapter: "docusaurus"
  toc_selector: ".menu__list"
  content_selector: "article"
  delay: 1.5
  url_policy: "drop_fragment"
```

### 可选但值得考虑的字段

- `generated_from`: 初始入口 URL
- `generated_at`: 蓝图生成时间
- `notes`: 给用户写备注用

### 原因

PRD 确实说蓝图主要描述“抓什么”，但这不等于它只能存内容树。对 V1 来说，**可复现执行本身就是产品体验的一部分**。用户要能确信今天生成的蓝图，明天 fetch 出来的行为仍然可理解。

---

## 6. “蓝图不存编号”是对的，但还不够，需要同时定义编号算法

spec 里说编号不写入 YAML，fetch 时动态生成，这个方向我同意，和 PRD“删减节点后编号始终连续”是对齐的。

### 但当前缺的东西是

- 编号到底基于“树位置”还是“输出文件位置”
- 目录节点是否参与编号
- 父节点有 URL 且有 children 时如何表现
- 根层和子层是否固定两位数补零
- 编号上限和深度上限有没有规则

### 建议明确规则

- 只对“实际输出文件的节点”生成文件编号
- 无 URL 的纯目录节点不出文件，也不占文件编号
- 有 URL 的父节点照常出文件，排在其子节点前
- 每层编号固定两位，如 `00`、`01`
- 文件名编号来自树结构，不来自抓取成功与否

### 最后这一点很关键

如果抓取失败会导致后续编号整体重排，用户重新跑一次后 NotebookLM 的 source 名称就会漂移，不利于引用定位。建议“编号基于蓝图结构”，不是“基于成功抓到的页面集合”。

---

## 7. 文件命名规则还缺少冲突处理策略

PRD 要求文件名可读，这很好，但当前 spec 只说 `{编号前缀}_{slugified_title}.md`。

### 还需要补的规则

- 同名标题不同页面如何处理
- 标题为空如何处理
- slug 太长是否截断
- 非 ASCII 标题如何转写
- 文件系统不允许字符如何清洗

### 建议

- 文件唯一性应主要依赖编号，不依赖 slug 唯一
- slug 只做可读辅助，允许冲突
- 空标题用页面主标题回填；还拿不到时用 `untitled`
- 统一限制 slug 长度，避免过长文件名

### 原因

PRD 强调的是“编号即结构”，不是“slug 唯一标识”。文件命名设计也应该体现这个优先级。

---

## 8. `generator.py` 的主流程对“生成蓝图”理解还偏窄

当前流程是：

1. 请求入口页
2. 命中 preset 就用 selector
3. 没命中就要求用户传 `--toc-selector`
4. 解析 TOC
5. 输出 YAML

### 这里至少有四个薄弱点

- 没定义入口页 URL 规范化
- 没定义 TOC 链接去重
- 没定义目录节点标题来源
- 没定义解析结果的质量反馈

### 建议补充的 `gen` 输出行为

生成成功后应打印 summary：

- 命中哪个 adapter / 还是 fallback
- 发现多少个 TOC 节点
- 其中多少个可抓取 URL 节点
- 过滤了多少外链 / hash-only link / 重复链接
- 蓝图输出路径

### 原因

PRD 的 Step 1 本质上是“生成可裁剪蓝图”，不是“偷偷爬完再写个 YAML”。用户需要知道蓝图质量如何，才能放心进入 Step 2。

---

## 9. TOC 提取逻辑不能只靠“递归解析嵌套列表”

当前 spec 假设 TOC 就是一个嵌套列表，这对很多文档站成立，但不够稳。

### 现实里会遇到的问题

- 有些站点目录节点是 `div + a`，不是标准 `ul/li`
- 有些目录包含折叠按钮、icon、重复标题
- 有些站点会把当前页二级标题混在侧边导航里
- 有些页面会有顶部页内目录和侧边全站目录同时存在

### 建议

TOC 提取要拆成两层：

- 先定位“候选导航容器”
- 再在容器内提取“可导航链接树”

同时在 spec 里明确过滤规则：

- 页内锚点不要进蓝图
- 外链不要进蓝图
- 重复 URL 去重
- 相同文本但不同 URL 允许共存

### 原因

PRD 要的是“文档站目录结构”，不是“页面里所有看起来像目录的东西”。技术方案需要对这个边界写清楚。

---

## 10. `presets.py` 目前更像脆弱的 selector 表，不像真正的站点适配策略

这个点和前一版建议一致，但这里展开说得更完整一点。

### 当前问题

- 把“框架识别”和“提取逻辑”分开了
- 每个 preset 只有 selector，没有配套清洗策略
- 同框架不同主题差异很大，容易误识别

### 建议

把 `preset` 改成“适配器”概念，每个适配器负责：

- `detect()`
- `get_toc_root()`
- `get_content_root()`
- `clean_content()`

通用 fallback 再负责：

- 用户显式 selector 优先
- 通用内容区 fallback：`article` / `main` / `[role=main]`
- 通用 TOC fallback：只在用户明确传 `--toc-selector` 时启用

### 原因

PRD 强调“宁可少自动，也不要错误自动”。适配器模式更符合这个原则，因为可以更明确地承认“没命中就不要瞎猜”。

---

## 11. 正文提取策略写得太轻，和 PRD 的“干净优先”不匹配

现在 `extract_content` 的描述只有：

- 提取正文
- 黑名单删除噪音
- fallback 到 `article/main/[role=main]`

### 问题

这还不足以保证输出“适合喂 LLM”。因为噪音不只是大块导航栏，还有很多文档站常见的小型污染：

- 复制按钮
- 编辑此页
- 上一页 / 下一页
- 面包屑
- 页内自动 TOC
- 版本提示
- admonition 标题重复
- tabs / accordion 的隐藏内容结构

### 建议

在 spec 中明确：正文提取分成两步

1. 定位内容根节点
2. 对内容根节点做结构化清洗

清洗规则至少包括：

- 删除导航性块
- 删除分页块
- 删除“编辑此页”“反馈”等站点附加块
- 处理代码块语言标识
- 保留表格、列表、标题层级
- 规范化相对链接

### 原因

PRD 明确说“宁可少抓，也不要混入噪音”。那正文提取必须是一个显式重点，而不能只是 parser 里的附带步骤。

---

## 12. `markdownify` 只能是工具，不应该被当成内容质量保证

当前 spec 对 `converter.py` 的表述太轻了，容易给人一种“正文抓出来后交给 markdownify 就差不多”的感觉。

### 这里需要明确

- 真正决定质量的是转换前清洗
- Markdown 渲染规则要受控，而不是完全交给第三方默认行为

### 建议在 spec 里写清楚转换关注点

- 标题层级保留
- fenced code block 保留
- 表格不要轻易丢结构
- 链接保留 href
- 图片保留远程 URL
- 不要把整个页面压成难读的纯文本

### 原因

PRD 要输出的是“干净、有序、可裁剪的 Markdown 文件集合”，不是“HTML 被粗略转成 Markdown”。这两者标准差很多。

---

## 13. URL 规范化完全没有展开，这是一个明显缺口

这是当前 spec 里缺得最明显的基础问题之一。

### 如果不先定规则，会直接影响

- TOC 去重
- 文件稳定性
- 请求是否重复
- 相对链接解析
- 后续 sync 的映射

### 建议在 spec 中明确

- 所有 URL 先转绝对 URL 再处理
- 默认去掉 fragment
- query 是否保留要配置化，默认尽量丢掉 tracking query
- 只允许同域或同 base_url 范围内链接进入蓝图
- 标准化 trailing slash

### 原因

PRD 说的是“文档站 → 知识源”，不是“网页任意链接集合 → Markdown”。URL 边界不明确，会直接破坏蓝图质量。

---

## 14. `fetcher.py` 的流程过于线性，缺少中间文档模型

当前写法：

1. 读 YAML
2. DFS 编号
3. 请求页面
4. 提取正文
5. 转 Markdown
6. 写文件

### 这条链路短期能用，但后面会出问题

- 很难插入 metadata/frontmatter
- 很难插入 sync 比较逻辑
- 很难插入图片本地化
- 很难把“抓取失败”和“渲染失败”区分开

### 建议引入中间模型

- `BlueprintNode`
- `FetchedPage`
- `ParsedPage`
- `RenderedDoc`

### 原因

这不是“过度设计”，而是为了把 PRD 后续路线留出自然扩展点。

---

## 15. `fetch` 的执行语义还不够明确

当前缺少以下定义：

- 输出目录已存在时怎么办
- 同名文件是覆盖还是报错
- YAML 中删除了节点，旧文件是否清理
- 抓取失败后是否保留部分产物

### 建议明确策略

对于 V1，建议默认：

- 输出目录不存在则创建
- 已存在同名文件则覆盖
- 不自动删除目录里其他旧文件，除非显式 `--clean`
- 最终输出 summary，告诉用户成功多少、失败多少

### 原因

PRD 的工作流里用户会反复裁剪 YAML 再 fetch。如果没有明确定义输出目录行为，用户很容易得到混杂的新旧文件集合。

---

## 16. `--force` 这个设计当前是悬空的

PRD 里 V1 用 `--force` 代替未实现的 sync，但当前 spec 没说明它具体强制什么。

### 问题

如果 V1 根本没有本地缓存、内容哈希或跳过逻辑，那 `--force` 实际上没有意义。

### 建议二选一

- 要么 V1 先删掉 `--force`
- 要么定义清楚：默认如果目标文件已存在且源 URL 未变则跳过，`--force` 才会全量覆盖

### 原因

参数语义不清会直接伤害 CLI 可信度。PRD 是给命令行用户用的，这类细节不能含糊。

---

## 17. 错误处理策略过于偏爬虫，不够偏产品

当前写的是：

- 遇到 429/503 重试 3 次
- 失败就跳过
- 继续下一页

### 这更像通用爬虫默认策略，但和 PRD 场景不完全匹配

用户不是来跑海量任务的，而是来构造高质量知识源的。少几页内容，结果价值会明显下降。

### 建议

- 默认严格模式：有失败就返回非 0
- `--keep-going`：允许跳过失败页
- 区分错误类型：
  - 蓝图错误
  - 网络错误
  - 解析错误
  - 渲染错误
- 命令结束输出失败清单

### 原因

PRD 的目标是让用户信任输出结果，而不是“尽量多抓一些再说”。

---

## 18. 反爬策略应该再降级表述，不要让它看起来像 V1 重点

固定 UA、随机 sleep、简单重试这些可以有，但不建议在 spec 里写得像一个重要卖点。

### 原因

PRD 没把“复杂反爬”当核心价值。V1 主要面对的是公开文档站，真正关键的是：

- 不要误抓
- 不要脏抓
- 不要行为不稳定

### 建议

把这一节改成“基础礼貌抓取策略”，语气更准确：

- 合理 UA
- 限速
- 简单重试
- 明确超时

---

## 19. 缺少超时、并发、会话复用的策略说明

虽然 V1 不需要复杂优化，但还是要把基础抓取行为写清楚。

### 建议补充

- 使用单个 `httpx.Client`
- 明确请求超时
- V1 默认串行抓取
- 后续若要并发，需要保证输出顺序与编号稳定

### 原因

PRD 重视的是稳定和可控，不是极限速度。把串行作为默认实现反而更贴合产品阶段，但最好明确写出来。

---

## 20. 缺少“页面最终标题来源”的规则

文件名和 Markdown 内容都依赖标题，但 spec 里没有定义标题优先级。

### 建议

明确标题获取优先级：

1. 蓝图节点 title
2. 页面主标题 `h1`
3. HTML `<title>`
4. URL path 派生

### 原因

PRD 的输出文件名会被用户直接上传给 NotebookLM，标题稳定性很重要。

---

## 21. 还没有定义 Markdown 文件内是否写 metadata

PRD 把 frontmatter 放到 V2，这个没问题。但 V1 至少要明确“纯正文”还是“允许最小头部信息”。

### 我建议

V1 可以不写 YAML frontmatter，但建议至少支持一个可选最小注释头，例如：

```md
<!-- source: https://... -->
```

或者把这类信息只写进 manifest。

### 原因

PRD 后续会做 metadata/frontmatter。V1 即便不暴露给用户，也最好不要让 source-to-file 映射彻底丢失。

---

## 22. `models.py` 里的模型定义偏弱，不能只靠最简 Pydantic 结构

当前模型：

- `TocNode(title, url, children)`
- `Config(selector, delay)`
- `ProjectConfig(...)`

### 问题

- `selector` 语义不清
- 没有版本字段
- 没有验证规则
- 没有 URL 规范化后的字段

### 建议

模型层至少要承担：

- YAML 结构验证
- 字段默认值
- URL 合法性检查
- 递归 children 验证

### 原因

PRD 的蓝图是“核心数据结构”，那模型层就不该只是个轻量容器。

---

## 23. 缺少 manifest 设计，会让 V1.5 的 sync 接不上

虽然 PRD 说 V1 不做 sync，但当前 spec 完全没有为后续留最小痕迹。

### 建议

每次 `fetch` 后写一个轻量 manifest，例如 `.docslice-manifest.json`：

- blueprint hash
- fetch time
- source URL -> output file
- source content hash
- final url
- status

### 原因

这不会明显增加 V1 复杂度，但能让 V1.5 的增量同步不需要反向推断历史状态。

---

## 24. 测试设计不够贴合“产品承诺”

当前测试思路是：

- `test_generator.py`
- `test_fetcher.py`
- `test_parser.py`
- 3 个真实站点集成验证

### 问题

- 真实站点会变，回归不稳定
- 测试命名按模块，不是按用户承诺
- 缺少针对裁剪后编号连续性的显式测试

### 建议测试拆法

优先做 fixture 测试：

- 蓝图生成是否正确
- URL 规范化和去重
- TOC 层级提取
- 内容清洗
- Markdown 关键结构保留
- 编号生成
- 用户裁剪后的连续编号
- 重复标题 / 空标题处理

真实站点测试只做 smoke test，可选运行。

### 原因

PRD 的验收标准是“编号连续、内容干净、层级正确”。测试应该直接围绕这些目标组织，而不是按工程模块凑。

---

## 25. 验证标准本身还可以更贴近 PRD

当前验收标准有：

- 编号连续
- 内容干净
- 文件名可读
- 层级关系正确

这些没问题，但还少几个和 PRD 强相关的维度。

### 建议补充

- 用户裁剪 YAML 后重新 fetch，编号仍稳定连续
- 同一站点重复执行，输出结果应基本稳定
- 蓝图可被人类直接编辑，不会因为程序重写而破坏结构
- 输出文件可直接用于 NotebookLM，单文件粒度不过粗

### 原因

PRD 的卖点不只是“抓到内容”，而是“适合做知识源”。

---

## 26. 发布方式不该进入主关键路径

当前实施步骤里最后有“PyPI 发布 + Homebrew tap”，这可以做，但排序上不应该太早。

### 我建议的顺序

1. 本地 CLI 跑通
2. 蓝图结构定稿
3. fixture 测试稳定
4. 真实站点 smoke test 通过
5. PyPI / pipx
6. Homebrew tap

### 原因

PRD 的风险主要在产品体验，不在安装方式。先把质量稳定下来，再补分发。

---

## 27. `brew install` 目标和用户画像并不完全冲突，但不该作为 V1 成功标准

PRD 的用户是熟悉命令行的人，所以 `pipx install` 其实已经足够早期验证。

### 建议

把 Homebrew 从“目标发布方式”改成“后续便利分发方式”。

### 原因

这是节奏问题，不是能力问题。过早强调 brew，会占掉不必要的工程注意力。

---

## 28. 还缺少“日志与用户反馈”设计

PRD 的工作流有明显的人机协作过程：生成蓝图、手动裁剪、再抓取。所以 CLI 输出不能太黑盒。

### 建议

`gen` 和 `fetch` 都应有清晰 summary：

- `gen`: 识别到什么站点类型、找到多少节点、输出到哪里
- `fetch`: 总节点数、待抓取文件数、成功数、失败数、输出目录

### 原因

这直接影响用户能否信任和调试这条流程。

---

## 29. 缺少对“纯目录节点”的明确产品定义

PRD 示例里已经隐含存在“有标题但无 URL”的目录节点，但 spec 没把它当成一等概念。

### 建议

在模型和实现里都把它明确为一种合法节点类型：

- `title` 必填
- `url` 可空
- `children` 可有

并写清楚其行为：

- 会出现在蓝图中
- 不会直接抓取
- 会影响树结构，但不一定影响文件编号

### 原因

PRD 的核心价值之一就是保住层级语义，这种节点不能只是“刚好兼容”。

---

## 30. 还缺少对“入口页不一定是文档根页”的考虑

用户可能对某个子路径执行 `gen`，而不是整个文档站首页。

### 当前 spec 没说明

- 是只生成当前子树
- 还是尝试反推整个文档站
- base_url 如何确定

### 建议

V1 明确规则：

- `gen <url>` 默认只围绕该 URL 所在导航树生成蓝图
- `base_url` 是站点根，不是入口页完整路径
- `generated_from` 记录初始入口 URL

### 原因

这和 PRD 的“手动裁剪”并不冲突，反而更符合用户对局部文档切片的实际使用方式。

---

## 31. 还缺少对“文档内容中的相对资源”处理边界的说明

PRD 说 V1 图片不下载，只保留图片链接。但除了图片，还有：

- 相对超链接
- 锚点链接
- 下载链接

### 建议明确

- 页面内相对链接统一改为绝对链接
- 站内锚点链接保留或去掉 fragment，要给规则
- 非 HTML 资源链接保留原始 URL

### 原因

PRD 的目标是做高质量知识源，链接处理不能含糊。

---

## 32. 实施步骤需要重排，应该按“降低产品风险”的顺序来

当前阶段划分：

1. 脚手架
2. `gen`
3. `fetch`
4. 集成测试
5. 打包发布

### 建议改成

1. 蓝图模型和编号规则定稿
2. URL 规范化、TOC 提取、内容清洗 fixture
3. `gen`
4. `fetch`
5. 端到端 smoke test
6. 打包发布

### 原因

现在的顺序太偏“先搭命令再说”，而 PRD 的最大风险其实在蓝图模型和内容质量，不先定这两个，后面实现容易返工。

---

## 33. 结论：我认为最需要改的不是功能清单，而是技术路线的优先级

如果要压缩成一句话：

**当前 `tech_spec.md` 最大的问题，是把 V1 当成“一个能跑的 CLI 抓取器”来设计，而不是当成“一个可裁剪、可复现、适合构建 LLM 知识源的文档切片产品”来设计。**

### 我建议优先收敛的点

1. 蓝图要能支撑可复现执行，而不只是保存目录树
2. 编号、URL、标题、目录节点这些基础规则要先定死
3. 提取流程要拆成适配、提取、清洗、渲染，而不是一步到位
4. 默认行为要偏“严格、可控、可调试”，而不是偏“尽量继续跑”
5. 测试和验收要直接绑定 PRD 承诺，而不是主要绑定真实站点是否今天刚好没变

### 如果只保留一个最核心判断

**PRD 最怕的不是功能少，而是输出结果不稳定、不可解释。**  
所以技术实现路径应该把“可复现、可调试、干净、稳定”放在“多框架覆盖、安装分发、快速跑通”前面。

## 第二次建议（2026-03-25：针对新版 `tech_spec.md` 及 `claude_sug.md` 的复审）

这次我看完新版 `tech_spec.md` 和 Claude 写的 `claude_sug.md` 之后，结论和第一次有明显变化：

**新版 `tech_spec.md` 已经吸收了大部分真正重要的改进点，整体上比第一版更贴合 PRD。**  
尤其是下面这些变化，我认为方向是对的：

- 开头明确把 V1 定义成“范围约束 + 核心链路稳定”，而不是泛泛的技术选型
- 蓝图字段补充了 `version`、`generated_from`、`toc_selector`、`content_selector`
- 编号算法、纯目录节点、有 URL 的父节点这些关键规则都写清楚了
- URL 规范化、文件命名 fallback、输出目录行为、错误处理都比第一版具体很多
- 项目结构里把 `extractor.py` 独立出来，正文提取和 TOC 解析不再混在一起
- 测试也从“按模块笼统测试”变成了更贴近产品承诺的拆分

从 PRD 角度看，这些修改已经把技术方案从“一个能跑的抓取 CLI”明显拉近到了“一个可裁剪、可复现、适合构建 LLM 知识源的文档切片工具”。

### 对 Claude 反驳点的重新判断

这次 Claude 不接受的那些点，绝大部分我认为他说得有道理，而且在当前 PRD 阶段，比我第一次建议更合适。

#### 1. `manifest` 不进入 V1：我认同 Claude

我第一次建议在 V1 写一个 `.docslice-manifest.json`，为 V1.5 的 sync 铺路。现在看，Claude 的反驳更符合 PRD：

- PRD 已经明确说 V1 不做 `sync`
- PRD 强调“不为未来的功能预埋复杂度”
- manifest 一旦落地，就不只是“留个痕迹”，而是会引入 schema、状态一致性、损坏恢复这些隐性复杂度

所以这个点我收回。V1 不做 manifest 是合理的。

#### 2. 四层中间模型不进入 V1：我基本认同 Claude

Claude 对这个点的反驳也是成立的。当前新版 spec 已经通过拆出 `extractor.py` 解决了最关键的职责问题，而没有继续把 `fetch` 抽成多层中间对象。

这对 V1 是合适的，因为：

- 当前主链路并不复杂
- 过早设计 `FetchedPage / ParsedPage / RenderedDoc` 这种中间模型，确实有较高概率接口定错
- PRD 当前需要的是“稳定主链路”，不是“提前建未来扩展框架”

不过我仍保留一个轻微意见：即使不引入显式中间模型，代码实现里也最好维持清晰的阶段边界，避免 `fetcher.py` 重新长成一个全能函数。

#### 3. 不新增 `docslice validate` 命令：我部分认同 Claude

我认同 V1 不需要多一个子命令。Claude 这里的取舍是合理的，因为：

- Pydantic 已经覆盖了基础结构校验
- 用户的主路径确实是“编辑 YAML 后直接 `fetch`”

但我还是保留一个判断：**验证能力本身仍然重要，只是不一定要做成独立命令。**

更好的 V1 做法是：

- `fetch` 在真正开始网络请求前，先完成蓝图预校验
- 把结构问题、重复 URL、缺失必要字段这类问题一次性报出来

这样既不增加命令面，又能保住 PRD 里“手动裁剪后流程可控”的体验。

#### 4. `url_policy` 不配置化：我认同 Claude

这个点我现在完全同意 Claude。

对于文档站这个场景，当前新版 spec 里写死的规则已经足够合理：

- 去 fragment
- 去 tracking query
- 规范 trailing slash
- 只保留同域链接

这些规则在 V1 里硬编码，比暴露成 YAML 配置更符合“渐进增强”和“少暴露无必要复杂度”的原则。

#### 5. 默认继续执行，不引入 `--keep-going`：我认同 Claude

Claude 在这个点上的判断更贴近真实使用场景。

新版 `tech_spec.md` 当前的策略是：

- 失败页记录下来
- 继续抓取剩余页面
- 结束时输出失败清单
- 有失败则 exit code 非 0

这个组合已经同时满足：

- 用户不会因为一页失败就失去其他产出
- 用户又不会误以为任务完全成功

所以这里没必要再额外加 `--keep-going`。

#### 6. 不做完整适配器类继承体系：我部分认同 Claude

我同意 V1 不需要上抽象基类、多态体系、完整 adapter class hierarchy。

Claude 给出的 `dataclass Preset` 方案在当前阶段是够用的，前提是它至少包含：

- `detect`
- `toc_selector`
- `content_selector`
- `noise_selectors`

也就是说，我同意“不要过度工程化”，但仍然不建议退回成“只有 selector 字符串表”的最薄配置层。  
如果 preset 仍然保留检测逻辑和清洗差异表达，那它就已经足够支撑 V1。

#### 7. 不拆成 10 个模块：我认同 Claude

这个点我也接受 Claude 的判断。

第一次我建议把模块进一步拆细，核心担心是职责混乱。现在新版 spec 已经做了最关键的拆分：

- `parser.py` 负责 TOC
- `extractor.py` 负责正文提取和清洗
- `converter.py` 负责 Markdown 转换

这已经足够。继续拆成更多模块，对 V1 的收益不大，反而可能让接口提前僵化。

### 我现在仍然保留的两个轻微意见

虽然大部分分歧我都接受 Claude 的判断，但还有两个点我觉得仍值得保留，只是表达方式要更收敛：

#### A. 不一定要有 `validate` 命令，但 `fetch` 前预校验要尽量完整

我不再坚持新增命令，但依然建议新版 spec 在 `fetch` 流程里明确写一句：

- 在开始抓取前，先完成蓝图合法性校验和 URL 去重检查

这样能减少用户在抓到一半时才发现蓝图错误的体验问题。

#### B. 不一定要有适配器继承体系，但 preset 不应退化成纯 selector 表

我不再坚持抽象类设计，但仍建议保留 Claude 现在这类结构化 preset 方向，而不要为了“简单”把 preset 压成一个只有 CSS selector 的静态映射。

原因还是一样：PRD 强调“宁可少自动，也不要错误自动”，那检测逻辑和清洗差异表达最好还是要有位置承载。

## 第二次建议结论

如果只看当前这版新版 `tech_spec.md`，我的判断是：

**它已经进入“可以开始认真实现 V1”的状态了。**

和第一次相比，我现在不再主张继续为 V1.5 提前铺更多结构。当前更合理的策略是：

1. 保持现有范围收敛，不额外引入 manifest、中间模型、独立 validate 命令
2. 按现在 spec 先把 `gen -> 手动裁剪 -> fetch` 主链路做扎实
3. 把注意力集中在输出质量、编号稳定、蓝图可复现、fixture 测试这几个真正贴合 PRD 的点上

如果只保留一句结论：

**Claude 这次拒绝的大部分点是合理拒绝；新版 `tech_spec.md` 的取舍，已经比我第一次建议时更符合这份 PRD 的阶段目标。**

## 第三次审评（2026-03-27：针对 agent-ready 技术文档的最终收口）

这次审评的标准和前两次不一样。前两次主要在看技术路线是否贴合 PRD；这一次看的是：

**`tech_spec.md` 是否已经足够清晰、足够决策闭环，可以让任何 AI agent 直接据此实现，而不偏离我们设计的初衷和本意。**

我的结论是：

**在这次重构之后，`tech_spec.md` 已经基本进入“agent-ready”的状态了。**  
它现在比之前更像一份“权威行为规范 + 实施建议”的技术文档，而不是一份把产品规范、Git 流程、环境操作、开发习惯混在一起的超长开发手册。

### 这次重构做对了什么

我认为最关键的改进有四个：

#### 1. 文档定位终于清楚了

现在的 `tech_spec.md` 明确声明自己是：

- V1 的权威实现依据
- 任何 AI agent 实现时的行为约束来源
- 规范优先，而不是“顺手写点开发步骤”

这件事很重要。因为如果一份技术文档不先声明“哪些内容是产品契约，哪些只是实现建议”，AI agent 很容易把边角操作步骤也当成必须遵守的产品要求。

#### 2. 规范和实施建议被分开了

这是我这轮最看重的变化。

之前的文档最大问题之一，是把下面几类内容混在一起：

- 产品行为定义
- 数据结构
- 错误处理
- Git 工作流
- conda 环境操作
- 分支命名
- commit 节奏

这对人类开发者还可以靠经验分辨轻重，但对 AI agent 并不友好。  
现在把前半部分收敛成行为规范，把后半部分降级为实施建议，这是正确方向。

#### 3. 影响输出结果的规则，基本都被写成唯一决策了

现在这版文档里，绝大多数会影响最终输出的关键点，都已经不是“模糊建议”，而是明确规则，例如：

- 蓝图字段和顺序
- `url` 在蓝图中存相对路径
- 编号算法如何处理纯目录节点和抓取失败
- `gen` 与 `fetch` 的参数优先级
- TOC 去重规则
- `fetch` 的退出码语义
- `source` 注释写最终跳转后的 URL
- 输出目录不自动清理旧文件

这些地方一旦不写死，不同实现者就很容易各做各的。现在大部分已经被固定下来，这对“可复现实现”非常关键。

#### 4. 文档更贴近 PRD，而不是贴近实现者个人习惯

这版的重心终于回到了 PRD 最关心的事情：

- `gen -> 手动裁剪 -> fetch`
- 编号即结构
- 干净优先
- 少自动、少误判

这说明文档已经不再是“某个开发者自己的施工顺序记录”，而是开始真正服务产品定义。

### 这版文档还剩下什么风险

虽然我认为它已经基本 agent-ready，但如果用“任何 AI agent 都应能无偏差实现”的标准去看，还是有几个需要保持警惕的点。

#### 1. TOC 提取的“支持边界”虽然写了，但仍然是最容易被实现者高估的地方

文档现在已经写明 V1 只支持标准导航列表结构，这是对的。  
但真正实现时，AI agent 很可能还是会冲动地去“顺手兼容” `div + a`、更复杂的 DOM 结构，或者自动猜哪个是 TOC。

所以这里我仍然建议实现时严格守住一点：

- 没有匹配到明确 `toc_selector`
- 或者命中后提不出可用节点

就应该直接失败，不要偷偷扩大能力边界。

这个风险不在文档本身，而在实现者容易自作聪明。

#### 2. “正文干净”依然是质量最高风险点

文档已经把提取和清洗规则写得比之前清楚很多，但正文质量依然是最难稳定的部分。  
尤其是以下内容，最容易在实现时被做得“勉强可用但不够干净”：

- 页内目录残留
- 复制按钮残留
- 前后页导航残留
- admonition / tabs 结构被破坏
- 表格退化过度

这意味着：**真正决定 V1 成败的，不是 CLI 框架，不是蓝图模型，而是 extractor + converter 的质量。**

这和 PRD 完全一致，所以后续实现时，测试和人工抽样都要把注意力集中在这里。

#### 3. 输出目录不自动清理旧文件，这个决策是合理的，但用户理解成本高

文档现在已经把这件事写清楚了，这比之前好很多。  
但它天然有一个用户侧风险：

- 用户裁剪蓝图后重新 fetch
- 目录里旧文件还在
- 用户误以为这些旧文件也是本次有效产物

也就是说，这个决策本身没问题，但实现时一定要把 summary 和文档提示写得足够直白。否则技术上没错，产品体验上仍然可能让人误解。

#### 4. 退出码和失败语义虽然收敛了，但 CLI 输出文案要同样稳定

这版文档已经把退出码定得比较清楚了，这很好。  
但对 AI agent 来说，还有一个容易被忽略的点：stdout / stderr 的 summary 文案也应该尽量稳定、结构化。

原因很简单：

- 人类用户要快速知道发生了什么
- 后续如果别的 agent 想包一层自动化，也需要可预测输出

我不建议 V1 为此上 JSON 输出之类的额外功能，但建议实现时让 summary 模板尽量固定，不要自由发挥。

### 这版文档现在是否足够让 AI agent 实现

我的判断是：

**足够了。**

更准确地说：

- 它已经足够让一个合格的 AI agent 在不新增产品决策的前提下完成 V1 实现
- 它也已经足够约束一个 AI agent 不要把 V1 做歪成“通用网页抓取器”或“半成品同步工具”

这正是这次收口最重要的目标。

### 如果还要继续优化，我只建议优化表达，不建议再改范围

到这个阶段，我不建议继续往 `tech_spec.md` 里塞更多能力或更多未来铺垫。  
后续如果还要改，我只建议做两类轻微优化：

#### A. 继续压缩重复表述

如果文档里某些规则在不同章节重复出现，可以后续做少量去重，让实现者更容易定位唯一权威描述。

#### B. 给测试章节补一个更显式的“验收矩阵”

现在测试和验收已经够用了，但如果未来真要交给多个不同 agent 并行实现，一个更表格化的“规则 -> 对应用例”映射会更稳。不过这已经属于增强项，不是当前缺口。

## 第三次审评结论

如果要用一句话概括我这次的判断：

**这版 `tech_spec.md` 已经从“技术方案”进化成“可执行规范”，可以作为 V1 的直接实现依据。**

它已经足够清晰，也已经足够贴近 PRD，不需要再做新的范围级修改。  
接下来的风险主要不在“文档是否缺设计”，而在“实现时是否严格尊重文档里已经写死的边界”。
